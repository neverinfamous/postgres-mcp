# Unreleased

## Added
- **Zod Schema Hardening**: Tightened `inputSchema` definitions for 23 parameter-less tools from permissive `z.object({})` to `z.object({}).strict()` to prevent arbitrary LLM argument passthrough.
- **Vector Error Handling**: Replaced generic `Error` throws with structured `ValidationError` classes in `pg_vector_cluster` and `pg_vector_dimension_reduce` for correct error categorization.
- **Audit Token Estimates**: Every audit entry now includes `tokenEstimate` (~4 bytes per token) computed from the serialized tool response. Gives agents and users historical visibility into token burn-rate per tool call.
- **Audit Read Logging (`--audit-reads`)**: Opt-in `--audit-reads` / `AUDIT_READS=true` flag enables audit logging for read-scoped tool calls. Read entries use compact format (omit `args`, `user`, `scopes`) for ~100 byte entries. Write/admin tools are always logged regardless of this flag.
- **Audit Log Rotation (`--audit-log-max-size`)**: Size-based log rotation via `--audit-log-max-size <bytes>` / `AUDIT_LOG_MAX_SIZE` (default: 10MB). When the log exceeds the limit, it rotates to `.1` (keeping only 1 rotated file). Prevents unbounded JSONL growth.
- **Audit Resource Session Summary**: `postgres://audit` resource now includes a `summary` block with `totalTokenEstimate`, `callCount`, `topToolsByTokens` (top 5 tools by aggregate token cost), and a human-readable `note`. Gives agents session-level token consumption visibility without parsing individual entries.
- **Inline Token Estimates**: All tool responses now include `_meta.tokenEstimate` in `content[].text` — gives agents zero-cost burn-rate feedback per turn. Estimate is `Math.ceil(Buffer.byteLength(json, 'utf8') / 4)`. Injected into text payload only — `structuredContent` remains schema-pure. Code Mode responses include `tokenEstimate` in `metrics`.
- **Audit Log**: JSONL audit trail for write/admin tool invocations with OAuth identity, timing, and outcome. Enable with `--audit-log <path>` or `AUDIT_LOG_PATH`. Use `--audit-log stderr` for Docker/K8s deployments where container logs pipe to the orchestrator. Async-buffered writes ensure zero performance impact on tool execution.
- **Audit Redaction**: `--audit-redact` / `AUDIT_REDACT=true` omits tool arguments from audit entries for sensitive environments.
- **Audit Resource**: `postgres://audit` resource returns the 50 most recent audit entries as JSON — gives agents read access to the audit trail without parsing files.
- **Audit Backup Snapshots**: Pre-mutation DDL snapshots for destructive/write tools (`pg_drop_table`, `pg_truncate`, `pg_vacuum`, `pg_migration_apply`, etc.). Enable with `--audit-backup` / `AUDIT_BACKUP=true`. Snapshots stored as JSON alongside the audit log with automatic retention cleanup (`--audit-backup-max-age`, `--audit-backup-max-count`). Optional data capture with `--audit-backup-data`. Non-throwing design — snapshot failures never block tool execution.
- **Audit Backup Tools (3)**: `pg_audit_list_backups` (list snapshots with filters, read scope), `pg_audit_restore_backup` (restore snapshot DDL+data in a transaction with dry-run support, admin scope), `pg_audit_diff_backup` (compare snapshot DDL against live schema to detect drift, read scope). All 3 return structured error when backup is not enabled.
- **Audit V2 — Non-Destructive Restore (§2)**: `pg_audit_restore_backup` now accepts `restoreAs` parameter to restore snapshots to a side-by-side table instead of overwriting live data. Enables safe cherry-pick or swap workflows without risking production tables.
- **Audit V2 — Semantic Diffing with Volume Drift (§3)**: `pg_audit_diff_backup` now includes `volumeDrift` in output — compares current row count and table size against snapshot metadata, surfacing data volume changes alongside schema drift.
- **Audit V2 — Code Mode Audit Coverage (§1)**: All Code Mode sandbox tool calls now pass through the `AuditInterceptor`, generating audit entries and pre-mutation snapshots. Previously, `pg_execute_code` operations bypassed the audit trail entirely. ~2ms overhead per inner tool call.
- **Audit V2 — Snapshot Compression + Async Writes (§4)**: Snapshots now use gzip compression (`.snapshot.json.gz`) and write asynchronously with fire-and-forget pattern. `BackupManager.flush()` added for graceful shutdown. Backward-compatible: reads both `.gz` and legacy `.json` files. New `--audit-backup-max-data-size` CLI flag (default 50MB) caps data capture for large tables.
- **Audit V2 — Safe Restore Workflow Prompt (§6)**: New `pg_safe_restore_workflow` prompt (20 total) — 6-step playbook guiding agents through snapshot identification, drift inspection, dry run, side-by-side restore, comparison, and merge/replace.
- **Audit V2 — Resource Enrichment POC (§7)**: `postgres://vacuum` resource now includes `suggestions[]` with severity-tagged, threshold-based actionable recommendations (dead tuple ratios, never-vacuumed tables, wraparound urgency). New `src/utils/resource-suggestions.ts` utility designed for extension to other resources.
- **Transport-Agnostic Auth**: Added `src/auth/transport-agnostic.ts` with `createAuthenticatedContext()`, `validateAuth()`, and `formatOAuthError()` — transport-agnostic auth utilities usable by Streamable HTTP or future transports without Express coupling. `middleware.ts` re-exports shared types from the new module.

### Fixed
- **Cron tools error standardization**: Fixed P154 structured error leaks in `pg_cron_job_run_details` and `pg_cron_cleanup_history` by adding missing `code: "VALIDATION_ERROR"` and `category: "validation"` fields to manual input validation checks. Added missing `error` field to `CronCleanupHistoryOutputSchema`.
- **Cron tools error boundaries**: Wrapped `pg_cron_create_extension` handler in `try/catch (error: unknown)` returning `formatHandlerErrorResponse()` to prevent unhandled database permission exceptions from bubbling up as raw MCP `-32603` errors.
- **Cron tools Split Schema validation leaks**: Fixed Zod validation leaks where missing required parameters (e.g., `schedule` in `pg_cron_schedule`) or wrong-type primitive parameters (e.g., passing `"abc"` to `days` in `pg_cron_cleanup_history`) triggered raw MCP `-32602` validation errors instead of structured `{success: false, error: ...}` responses. Added `.optional()` to required fields and `z.preprocess(coerceNumber, z.number().optional())` to numeric fields in the MCP-visible Split Schema `Base` definitions, ensuring validation logic is properly decoupled and evaluated during the handler's internal `try/catch` execution block where errors are correctly caught and transformed.
- **Cron tools Zod leaks (`pg_cron_job_run_details`, `pg_cron_cleanup_history`)**: Fixed a Split Schema architectural bug where `z.preprocess()` on the base MCP input schema accidentally masked `.optional()` parameters, causing `zodToJsonSchema` to expose them to LLMs as rigidly required fields. Removed the outer `.preprocess()` logic from the exposition-only Base schemas, preserving accurate metadata while keeping strict numeric coercion securely encapsulated within the handlers.
- **Cron tools limit coercion (`pg_cron_list_jobs`)**: Extracted inline Zod schemas into `schemas/cron.ts` to adhere to the Split Schema pattern. Fixed a Zod validation leak where string limit inputs like `"abc"` threw raw `NaN` validation exceptions instead of safely defaulting to 50 via `coerceNumber`.
- **Vector tools Zod leaks (`pg_vector_search`, `pg_hybrid_search`, `pg_vector_insert`, `pg_vector_batch_insert`, `pg_vector_distance`)**: Fixed bugs where missing required array or string parameters triggered raw MCP `-32602` validation errors from the SDK instead of structured `{success: false, error: ...}` responses. Added `.optional()` and `z.string()` to the MCP-visible Split Schema `Base` definitions and moved rigorous validation into the tool handlers.
- **Vector tools Query leaks (`pg_vector_batch_insert`, `pg_vector_index_optimize`)**: Fixed an issue where missing table and column string parameters bypassed Zod base-schema defaults and triggered raw DB `Invalid identifier` query engine exceptions instead of structured `{success: false, error: ...}` responses. Added explicit internal validations against empty strings to both handlers.
- **Schema numeric coercion leak**: Fixed a Zod validation leak where `coerceNumber()` swallowed invalid types (returning `undefined` for `NaN`), allowing inputs like `start: "abc"` to silently succeed without throwing a `ZodError`. `coerceNumber()` now returns the original non-numeric value so typed handlers can correctly emit `{success: false, error: "Validation error: Expected number..."}` instead of bypassing constraints.
- **Schema Split Schema consistency**: Resolved Split Schema architectural violations in `schema-mgmt.ts`. Removed inner `z.preprocess()` wrappers from `CreateSequenceSchemaBase`, `ListSequencesSchemaBase`, `ListViewsSchemaBase`, and `ListFunctionsSchemaBase` — replacing them with `z.unknown().optional()` — to ensure parameter metadata is preserved in the JSON Schema exposed to the MCP client, while strictly enforcing numeric types with `coerceNumber` locally within the decoupled Handler schemas.
- **Backup Split Schema bugs (`pg_audit_restore_backup`, `pg_audit_diff_backup`)**: Fixed P154 error standard violations where Zod validation and domain errors were manually returned as `{success: false, error: ...}` without `code` or `category` fields. Handlers now throw structured errors to integrate seamlessly with `formatHandlerErrorResponse()`, and their output schemas officially extend `ErrorResponseFields.shape`.
- **In-place Restore TYPE logic failure**: Fixed a bug in `pg_audit_restore_backup` where in-place restore (`confirm: true`) failed with "relation is not a sequence". The tool erroneously generated a `DROP SEQUENCE` statement for tables because it naively matched the object type from the first `CREATE` keyword (usually `CREATE SEQUENCE` for `SERIAL` columns in schemas). It now generates a robust cascade of `DROP TABLE`, `DROP VIEW`, `DROP MATERIALIZED VIEW`, and `DROP SEQUENCE` statements to safely guarantee target object destruction before recreation.
- **Backup Split Schema adherence**: Replaced inline Zod schemas in `pg_audit_restore_backup` and `pg_audit_diff_backup` with exported Split Schema definitions (`AuditRestoreBackupSchema`, `AuditDiffBackupSchema`) from `schemas/backup.ts`.
- **Payload bloat in `pg_audit_list_backups`**: Added `limit` parameter (default 50) and `truncated` boolean, preventing unbounded JSON array sizes for databases with heavy snapshot activity.
- **In-place Restore Serial failure**: Restoring tables with `SERIAL` columns via `pg_audit_restore_backup` no longer fails with `relation does not exist` on the sequence default. `pg_dump_table` now explicitly queries `pg_depend` for explicitly-owned sequences and prepends `CREATE SEQUENCE IF NOT EXISTS` commands to the snapshot DDL before the `CREATE TABLE` statement.
- **Backup Split Schema bugs (`pg_create_backup_plan`, `pg_copy_export`, `pg_backup_physical`)**: Fixed a Split Schema pattern violation where `z.preprocess()` was used directly inside `inputSchema`, causing parameter metadata (like `.optional()`) to be stripped from JSON Schema generation. Moved coercion to parser-specific schemas and replaced base schemas with strict primitives.
- **Code Mode backup namespace parity**: Added `listBackups`, `diffBackup`, and `restoreBackup` intuitive aliases to the `pg.backup` Code Mode namespace in `maps.ts`, resolving `TypeError: pg.backup.listBackups is not a function` during `pg_execute_code` script executions.
- **Admin tool Split Schema bugs (`pg_terminate_backend`, `pg_cancel_backend`, `pg_append_insight`)**: Fixed Split Schema alias visibility bugs where `pg_terminate_backend` and `pg_cancel_backend` incorrectly marked both `pid` and its alias `processId` as required in `TerminateBackendSchemaBase` by replacing `z.preprocess` with standard `z.number().optional()`, fixing MCP client compatibility errors. Also added the missing `text` alias to `AppendInsightSchemaBase` and wired it through to the handler's schema to ensure the LLM interface correctly accepts the documented alias.
- **Admin tool Zod leaks (`pg_append_insight`, `pg_set_config`)**: Fixed a bug where `pg_append_insight` returned a raw JSON array instead of a structured handler error on empty parameters by importing and using `formatHandlerErrorResponse()` in the catch block. Fixed a Split Schema pattern violation in `pg_set_config` where `value` was not marked as `.optional()` in `SetConfigSchemaBase`, causing the MCP SDK to throw raw `-32602` errors when aliases were used instead of passing through to the handler's structured validation.
- **Admin tool Zod leaks (`pg_reindex`)**: Replaced `z.enum()` with `z.string()` in `ReindexSchemaBase` and `ReindexSchema` so invalid or missing targets reach the handler's `try/catch` instead of triggering raw MCP `-32602` errors. Added `z.string().refine()` for enum validation inside the handler schema.
- **P154 Structured Error Leak**: Added missing `code: "VALIDATION_ERROR"` and `category: "validation"` fields to manual input validation checks in `pg_query_plan_compare` and `pg_partition_strategy_suggest` to ensure full structural adherence to the `ErrorResponse` schema payload pattern.
- **Code Mode Split Schema**: Fixed an issue where `pg_execute_code` with empty parameters threw a raw MCP `-32602` validation error instead of a structured JSON response. Refactored `src/adapters/postgresql/tools/codemode/index.ts` to use the Split Schema pattern (with `ExecuteCodeSchemaBase`) and wrapped the handler in a `try/catch` block with `formatHandlerErrorResponse()`.
- **Test script hangs**: Resolved infinite hangs when running `reset-database.ps1` under Docker on Windows. Script now correctly defaults the database password to `postgres` (preventing `psql` interactive prompts), pipes massive `DO $$...` DDL blocks through stdin instead of `-c`, and uses precise container inspection to bypass array evaluation mismatches in PowerShell.
- **Instruction filtering regression**: `CORE_INSTRUCTIONS` constant embedded a static `## Help Resources` section with `postgres://help/{group}` that bypassed level/filter gating. Removed static section; help pointers are now generated dynamically by `buildHelpPointers()` at `standard`+ level. Essential level gets a minimal `postgres://help` pointer without group listing. Fixes 3 test failures in `test-filter-instructions.mjs`.
- **Test script DB auth**: Integration test scripts (`test-instruction-levels.mjs`, `test-filter-instructions.mjs`) timed out when `POSTGRES_PASSWORD` was not in the shell environment. Added Docker-default fallback (`postgres`/`postgres`/`postgres`) when neither `POSTGRES_CONNECTION_STRING` nor `DATABASE_URL` is set.
- **Test expectation for codemode auto-injection**: Updated test expectations to match the existing `parseToolFilter()` design where codemode is auto-injected in whitelist mode (e.g., `--tool-filter core` includes codemode).
- **`volumeDrift.rowCountSnapshot/Current` always undefined**: `pg_class.reltuples::bigint` is sent as a string by the pg driver. The `typeof row_count === "number"` guard always evaluated `false`, so `rowCount` was never captured in snapshots and never compared in diffs. Fixed with `parseInt(String(...))` in both `backup-manager.ts` (snapshot capture) and `audit-backup.ts` (diff comparison). `volumeDrift.rowCountSnapshot` and `rowCountCurrent` now populate correctly when the table has been analyzed.
- **`volumeDrift` reports `-1` as current row count on unanalyzed tables**: PostgreSQL uses `reltuples = -1` as a sentinel meaning "statistics not yet collected". Previously this surfaced literally as `rowCountCurrent: -1` with summary text like `"Row count changed from 5 → -1"`. Now treated as `undefined` (omitted from output, no misleading summary generated). Affects tables that have never been `ANALYZE`d after creation.
- **`describeTable` returns stale schema after DDL**: Metadata cache (30s TTL) was not invalidated after `ALTER TABLE`, `CREATE TABLE`, `DROP TABLE`, `TRUNCATE`, or index DDL. Added DDL-aware cache invalidation in `executeWriteQuery()` — detects DDL statements via regex, extracts the target table name, and clears the affected `describe:*`, `list_tables`, and `all_indexes` cache entries.
